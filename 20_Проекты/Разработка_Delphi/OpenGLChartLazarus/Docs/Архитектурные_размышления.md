# OpenGLChartLazarus: архитектурные размышления

Этот документ фиксирует идеи до превращения их в окончательный API.

## Страница как контейнер

Страница графика не просто прямоугольная область. Она должна быть контейнером для вложенных компонентов:

- осей;
- серий;
- курсоров;
- подписей;
- вспомогательных панелей;
- вложенных страниц или областей.

Страницы должны уметь менять размер, пересчитывать дочерние объекты, примагничиваться друг к другу и сохранять устойчивое расположение при изменении размера окна.

## Две системы размеров

Для размещения нужны два представления:

- пиксельное: фактический прямоугольник на экране;
- относительное: float-координаты, например доли от родителя.

Float-размещение приоритетно при загрузке. Это важно, потому что при чтении конфигурации OpenGL-контекст может еще не существовать, а пиксельные размеры окна или страницы могут быть неизвестны или слишком малы. Относительная геометрия не зависит от текущего размера окна, поэтому позволяет восстановить компоновку заранее, а пиксели вычислить позже.

При изменении размера экрана страницы и дочерние компоненты должны пересчитывать пиксельные bounds из float-геометрии. Это дает масштабируемость без потери логической компоновки.

## Режимы координат

Объект может рисоваться в разных системах координат:

- в координатах родителя;
- в координатах окна;
- в координатах данных графика;
- в смешанном режиме, например X в данных, Y в пикселях.

Поэтому у объекта должен быть явный `DrawMode` или `CoordinateMode`. Рендер не должен угадывать, как трактовать координаты.

## Bounds и hit-test

Каждый объект должен уметь:

- вычислить пиксельные bounds;
- вычислить логические bounds;
- сказать, попадает ли точка мыши в объект;
- отдать bounds для автоподбора масштаба, если объект участвует в диапазоне данных;
- пересчитать внутреннюю геометрию при изменении родителя.

Это не только для рисования. Такие расчеты нужны для выбора объектов, drag/resize, привязки страниц, подсказок и редакторов.

## Frame listeners

Реакции на пользовательские события не должны быть размазаны по объектам. Для этого нужен слой `frame listener`:

- listener получает события мыши, клавиатуры и изменения контекста;
- listener может включаться и выключаться в зависимости от режима;
- listener работает с объектами через базовый интерфейс, а не через знание конкретного класса;
- несколько listeners могут существовать одновременно, но активный контекст решает приоритеты.

Объекты дают базовый интерфейс: bounds, hit-test, coordinate transform, draw mode, selection state, children. Благодаря этому один frame listener может работать с разными объектами.
