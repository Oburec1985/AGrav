# Паттерн: EAV (Entity-Attribute-Value)

Этот подход был выбран для реализации **SQLSensorsDB** для обеспечения гибкости хранения характеристик оборудования.

## Суть идеи
Вместо того чтобы создавать сотни колонок для каждого уникального параметра датчика (напр. «Диапазон», «Частота», «Резьба»), мы храним характеристики в виде строк в отдельной таблице `Product_Specs`.

### Структура в нашем проекте:
1.  **Entity**: Ссылка на датчик (`product_id`).
2.  **Attribute**: Название параметра (`param_name`).
3.  **Value**: 
    - Текстовое значение (`param_value`) для отображения.
    - Числовое значение (`numeric_value`) для математических фильтров и поиска.

## Почему это качественное проектирование:
- **Масштабируемость**: Добавление нового типа датчика с уникальными параметрами не требует изменения схемы БД (`ALTER TABLE`).
- **Мощный поиск**: Мы можем строить динамические фильтры (слайдеры) по любым числовым атрибутам без ограничений жесткой схемы.

## Когда применять:
Когда количество возможных характеристик объекта велико, они постоянно меняются или заранее неизвестны (напр. каталоги сложной техники, медицинские карты).

*Пример из практики реализации SQLSensorsDB.*
