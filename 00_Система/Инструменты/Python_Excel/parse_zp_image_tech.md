# Технические Нюансы parse_zp_image.py

Документ фиксирует рабочие решения для распознавания зарплатных JPG, чтобы не потерять найденные оптимизации.
## Цель

Получить пары `ФИО -> ИТОГО к выплате` из изображений ведомостей и сохранить их в `result.csv` для последующего сопоставления с Excel.
## Геометрия Таблицы

Скрипт не распознает всю таблицу OCR. Сначала OpenCV ищет линии сетки:

- горизонтальные линии дают `y_coords`;
- вертикальные линии дают `x_coords`;
- близкие линии склеиваются через `merge_close_coords`.

В январских ведомостях каждый зарплатный блок является отдельной мини-таблицей из трех вертикальных линий:
```text
левая граница | разделитель | правая граница
```
Поэтому блоки группируются шагом 3 через `build_salary_blocks(x_coords)`. Это важно: скользящее окно с шагом 2 создает ложные блоки и узкие ячейки суммы шириной около 13 px.

## Строка Итоговой Выплаты

Целевая строка только одна:
```text
ИТОГО к выплате
```
Строка `СУММА` не используется как итоговая выплата, потому что это другое значение.

Для текущего формата ведомости строка `ИТОГО к выплате` находится на `row_idx = 20`. Координаты берутся из `y_coords[20]:y_coords[21]`. Если формат изменится, надо проверить лог `Найдено линий` и визуально сверить индекс строки.

## Распознавание Сумм (Оптимизация Апрель 2026)

Для денежных ячеек используется агрессивная многослойная стратегия `ocr_amount_cell`:

1. **Параллелизм**: Tesseract запускается в 10-12 потоков через `ThreadPoolExecutor`, одновременно проверяя все варианты обработки. Это дает ускорение в 3-4 раза.
2. **Масштабирование**: Ячейка увеличивается в **8 раз** (вместо 6) с интерполяцией `LANCZOS4`. Это критично для шрифтов высотой 14-16 px.
3. **Пул «Мнений» (Варианты предобработки)**:
   - **Otsu & Adaptive**: Базовые бинарные фильтры.
   - **Gamma 0.5 («Dark Side»)**: Принудительное затемнение серых пикселей. Решает проблему разрывов в цифрах `5` и `6`.
   - **Thick (Threshold 190 + Erode)**: Физическое утолщение линий для замыкания контуров.
   - **Erosion**: Утончение линий. Помогает, если цифры `3` и `8` сливаются в пятно.
4. **Гибридные движки**:
   - `--oem 1` (LSTM): Основной движок.
   - `--oem 0` (Legacy): Используется как контрольный. Он точнее на простых шрифтах, так как работает по шаблонам, а не по контексту.
5. **Весовое Голосование**: Каждому режиму присвоен вес (стандартные — 1.0, вспомогательные — 0.5-0.8). Побеждает число, набравшее максимальную сумму весов. Это исключает влияние случайных галлюцинаций Tesseract.
6. **Конфиг `digits`**: Используется встроенная в Tesseract конфигурация для чисел, дополненная строгим whitelist.
7. **Микро-обрезка (1 px)**: Удаляет линии сетки, которые Tesseract может принять за цифру `1` в конце числа.

## Умное Кэширование Геометрии

Чтобы не искать строку «ИТОГО к выплате» в каждом из 15-20 блоков (что медленно):
- Скрипт ищет её в первом блоке через нечеткий поиск.
- Найденный индекс (например, 20) кэшируется.
- В следующих блоках скрипт сначала проверяет именно этот индекс. Если сумма там правдоподобна, полный скан блока пропускается.
## Фильтр Мусора

Сумма принимается только если она проходит `amount_is_plausible`:
```text
50000 <= сумма <= 500000
```

Это отсекает явные ошибки OCR: миллионы, триллионы, единичные цифры и мусорные строки.

Опасный fallback “взять максимальное число из нижних строк” удален. Если основная ячейка `ИТОГО к выплате` не распознана, скрипт пишет `WARNING`, а не подставляет случайную сумму из строк `Аванс`, `Зарплата`, `СУММА` или `Баланс`.
## Сопоставление ФИО

`employees.txt` используется как словарь эталонных имен.

Порядок работы:
1. OCR читает заголовок блока.
2. Для ФИО сначала берется только левая ячейка первой строки блока; правая ячейка с месяцем не подмешивается в OCR.
3. Ячейка ФИО масштабируется по высоте строки, прогоняется через несколько предобработок (`otsu`, `adaptive`, `sharp`, `gray`) и OCR-режимов `psm 7/8/13` с русским whitelist.
4. Имя нормализуется: убираются месяц, год, знаки, лишние пробелы, но сохраняются точки и дефисы для различения инициалов.
5. **Весовое сопоставление**: Для каждого ФИО из `employees.txt` считается похожесть через `difflib.SequenceMatcher`. Если уверенность ниже 20%, имя считается ненайденным (`Unknown`), чтобы избежать ложной подстановки первого попавшегося сотрудника из словаря.
6. В CSV пишется ближайшее эталонное ФИО из `employees.txt`; позиционная подмена не используется как основной механизм.

Так Excel получает стабильный ключ, даже если OCR исказил заголовок.

## Перенос В Excel

`update_excel_template.py` является вторым этапом цепочки:

```text
JPG -> parse_zp_image.py -> result.csv -> update_excel_template.py -> *_updated.xlsx
```

CSV читается функцией `load_mapping_from_csv`. Поддерживаются имена файлов `result.csv` и `results.csv`.

Excel-файл выбирается функцией `resolve_excel_file`: сначала проверяется точный путь, затем выполняется поиск `.xlsx` по подстроке имени в `--search-dir`. В поиск не попадают временные файлы Excel `~$*` и уже созданные результаты `*_updated.xlsx`.

Сопоставление с Excel:

- CSV должен иметь колонки `ФИО;Значение`;
- имя из Excel нормализуется через `normalize_name`;
- сначала ищется точное совпадение нормализованных строк;
- затем допускается частичное совпадение через `startswith`;
- значение записывается в целевую колонку;
- исходный файл не перезаписывается, результат сохраняется как `*_updated.xlsx`, если явно не задан `--output`.

Колонки можно задавать буквами (`B`, `H`) или 1-based номерами (`2`, `8`).

Если в CSV есть дубли ФИО, используется последнее значение и выводится `WARNING`. Это важно проверять после распознавания, потому что дубли обычно означают ошибку сопоставления имени на этапе OCR.

## Батник Полного Цикла

`run_salary_pipeline.bat` является пользовательской обвязкой над двумя Python-скриптами:

```text
parse_zp_image.py -> result.csv -> select_excel_file.ps1 -> update_excel_template.py
```

Технические детали:

- запуск идет из каталога батника через `cd /d "%~dp0"`, поэтому картинки, `result.csv`, `employees.txt` и Excel ищутся рядом с инструментом;
- перед OCR задается вопрос `Run image parsing and rebuild result.csv? [y/N]`; Enter пропускает парсинг и использует уже готовый `result.csv`/`results.csv`;
- Python выбирается по очереди: `py`, затем `python`, затем запуск `.py` через файловую ассоциацию Windows;
- список Excel-файлов строится отдельным `select_excel_file.ps1` по подстроке, чтобы корректно работать с пробелами и русскими именами;
- если найдено несколько `.xlsx`, по умолчанию автоматически берется первый файл;
- интерактивный выбор номера доступен через ключ `-Interactive` у `select_excel_file.ps1`;
- если строка поиска не передана, `select_excel_file.ps1` использует дефолт `РСС`;
- путь передается обратно в `.bat` как короткий DOS-путь через `Scripting.FileSystemObject.ShortPath`, чтобы `cmd.exe` не ломал русские символы при `chcp 65001`;
- `update_excel_template.py` разворачивает короткий 8.3 путь обратно в длинный Windows-путь через `GetLongPathNameW`, потому что короткое имя `.xlsx` может выглядеть для `cmd` как `.XLS`, а `openpyxl` отвергает такие имена как старый Excel-формат;
- параметры листа и колонок передаются в `update_excel_template.py`;
- батник передает `--output` с тем же выбранным Excel-путем, поэтому файл обновляется на месте.
- батник передает `--value-divisor 1000`, поэтому суммы из `result.csv` записываются в Excel в тысячах.
## Логирование

`ocr.log` очищается при каждом запуске.

В боевом режиме лог должен оставаться коротким:
- окружение Python/Tesseract;
- файл и размер изображения;
- количество найденных линий и блоков;
- выбранная строка `ИТОГО к выплате`;
- найденные пары `ФИО -> сумма`;
- `WARNING`, если имя или сумма требуют внимания.

Временные подробные диагностические логи `[GEOM]`, `[SCAN]`, `[CELL]`, `[FALLBACK]` удалены из основного потока, чтобы лог не разрастался.
## Когда Меняется Формат Ведомости

Если ведомость получила другую сетку:
1. Проверить число `X` и `Y` линий в `ocr.log`.
2. Визуально определить, остается ли блок трехлинейным.
3. Проверить индекс строки `ИТОГО к выплате`.
4. При необходимости обновить `build_salary_blocks` и индекс `visual_expected_row`.
5. Не возвращать OCR всей таблицы: это медленно и нестабильно.
