# Разработка и Стандарты Кода (AGrav)

Данный документ содержит технические стандарты разработки, накопленные в ходе работы с проектами в репозитории AGrav.

## 🛠 Работа с кодом (Delphi 2010 + General)

### Кодировки
- Всегда использовать **UTF-8** для файлов с русским текстом.

### Стандарты именования
- **Локальные переменные**: префикс `l` (например, `lNum`).
- **Глобальные переменные**: префикс `g_` (например, `g_DBase`).
- **Поля классов (protected/private)**: префикс `f` или `fProperty`.
- **Члены классов (access)**: префикс `m_`.

### Управление памятью
- Стараться **не пересоздавать объекты**, а очищать данные в них (реализовывать методы `Clear`).
- В RunTime память лучше не выделять (создавай заранее при инициализации/переконфигурировании);
- В многопоточности: менять состояние потока только из самого потока; **не выделять память в RunTime** (внутри потока).

### Тестирование
- Каждая новая фича должна сопровождаться тестом (корректность + пример использования).

## 📂 Структура и инструменты
- **Каталог `cach`**: Вести план доработок (`plan.md`) и технические заметки (`notes_xxx.md`) индивидуально для каждого проекта.
- **Shared Utils**: Общие библиотеки хранить в `sharedUtils` (math, lists и др.).
- **Инструменты Excel**: Использовать `ExcelMcpPrj.exe` (специфический MCP сервер) согласно `tools_ru.md`.

## 🔍 Протокол отладки (Логирование) (Autonomous Debugging)

При поиске ошибок ИИ обязан внедрять временные логи согласно стандарту:

### 1. Формат лога (High-Res Log)
`[TIME: HH:mm:ss.SSS] [TID: 0x000] [TYPE: ENTER|LOG|EXIT|ERROR] [CONTEXT: ProcName, Line: NN] [DATA: var1=val, var2=val]`
- **Time**: Время до миллисекунд.
- **TID**: ID потока (обязательно для многопоточных систем).
- **Type**: Маркер входа/выхода для отслеживания "падений" внутри процедуры.
- **Context**: Имя процедуры и номер строки вызова.
- **Auto_<err>**: Признак, что автострока для поиска опред. ошибки. Авто строки можно удалить, когда ошибка найдена или точно находится  в другом месте

### 2. Принципы автономности
- **Хирургическое внедрение**: Логи добавляются только в подозрительные участки кода.
- **Потокобезопасность**: Запись в файл должна быть обернута в `Lock` / `CriticalSection` или использовать системный Thread-Safe Logger.
- **Самоочистка**: После того как участок кода признан исправным, ИИ **обязан удалить** внедренные строки лога, чтобы не загрязнять бизнес-логику.
- **Flush on Crash**: Использовать принудительную запись на диск перед потенциально опасными операциями.

### 3. Техника "Захвата падения"
Внедрять минимум две строки:
1. `ProcName: Start (L:120, params...)`
2. `ProcName: End (L:150)`
Если в логе есть Start, но нет End — место аварии локализовано.
