# Обслуживание и Очистка RouteRich AX3000

Документация по регулярному техническому обслуживанию и очистке памяти роутера RouteRich AX3000 (OpenWrt-based).

---

## 🧹 Процедура очистки (Cleanup)

При активной эксплуатации (установка пакетов, смена конфигов) в системе накапливается временный мусор. Безопасные шаги по его удалению:

### 1. Удаление списков пакетов (opkg lists)
Списки занимают место в RAM (`/tmp`) и могут дублироваться в `/var`.
```bash
rm -rf /var/opkg-lists/* /tmp/opkg-lists/*
```
*Последствие: после очистки команда `opkg update` скачает их заново.*

### 2. Очистка кэша LuCI (Web-интерфейс)
Иногда LuCI кэширует старые индексы и модули.
```bash
rm -rf /tmp/luci-modulecache/* /tmp/luci-indexcache*
```

### 3. Управление логами и бэкапами
- **Лишние бэкапы**: Старые копии конфигов следует удалять только после того, как настройки стабилизированы и проверены на всех ресурсах.
- **Итерационные бэкапы**: При проведении опытов использовать расширения `.v1`, `.v2`, `.stable`.

## 🛠 Конфигурация Сети и Обход Блокировок

Данный раздел содержит знания, полученные в ходе оптимизации Telegram, 4pda и WhatsApp (апрель 2026).

### 1. Распределение трафика (Podkop)
Для обеспечения скорости медиа-данных (фото, видео) рекомендуется разделять трафик:
- **Youtube_Discord (VPN)**: Сюда переносятся `telegram`, `meta` (WhatsApp/Instagram), `discord`, `youtube`. Это дает прямой доступ через туннель без посредничества прокси.
- **main (Proxy/DPI)**: Для обычных заблокированных сайтов (RuTracker и т.д.).

### 2. Устранение DNS-конфликтов
- **Zapret vs Podkop**: `zapret` (nfqws) может конфликтовать с DNS-перехватом Podkop, вызывая "зависание" сети. Если Podkop не выдает Fake-IP (198.18.x.x) для заблокированного домена, первым делом стоит временно отключить zapret (`/etc/init.d/zapret stop`).
- **IPv6 Leaks**: Некоторые сайты (напр. 4pda.to) имеют IPv6 адреса, которые идут в обход IPv4-туннеля. Решение: блокировать IPv6 через `dnsmasq` (`list address '/domain/::'`).
- **Hardcoded IPs**: Всегда проверяйте `/etc/hosts` на наличие старых жестких привязок IP, которые ломают логику "фейковых" адресов.

### 3. Правило "Increment & Snapshot"
**ОБЯЗАТЕЛЬНОЕ ПРАВИЛО ДЛЯ ИИ:**
При любых изменениях в сетевой конфигурации ИИ обязан:
1. Создать бэкап текущего рабочего конфига (`/etc/config/podkop.stable`).
2. Вносить изменения по одной итерации за раз.
3. Документировать в логе, какое именно изменение повлияло на работоспособность сервисов (например, Telegram Web).

## 🔍 Инструменты анализа

Для поиска "пожирателей места" в основной памяти (`/overlay`):

| Задача | Команда |
| :--- | :--- |
| Общий статус памяти | `df -h` |
| Поиск крупных файлов (>100КБ) | `find /overlay -type f -size +100k -exec du -h {} + \| sort -rh` |
| Вес конкретных папок | `du -sh /tmp /overlay /root` |

## ⚠️ Критические компоненты
**НЕ УДАЛЯТЬ без явной необходимости:**
- `/usr/bin/sing-box` (~28 MB) — ядро обхода блокировок.
- `/usr/bin/opera-proxy` — альтернативные методы проксирования.
- `/opt/zapret/` — файлы системы `zapret`.
- `/etc/config/` — основные настройки системы.
